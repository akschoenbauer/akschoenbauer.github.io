Challenge 4: A Google Search Page

New Attribute Use: In my challenge 4 code, I utilized a new attribute called colgroup. The colgroup tag allows a coder to apply styling and size guidelines to specific columns in a table. For example, you could use this attribute to change the background color of just one of the columns in a table. In my code, I used colgroup to set the widths of the two columns of the table I used to create the sub-links under the first Google serach result. I learned about what all can be done with this attribute and how to properly use it in my code through the W3Schools HTML colgroup Tag documentation page.

Challenge 5: Accessibility Concerns

Changes made: I added in some reflection on accessibility concerns that came up while I was creating my Google Search Page Replicate.

While working on Challenge 2, I began to wonder how a screen reader would approach the chessboard and chess piece code on this webpage. In lecture, it has been mentioned that screenreaders often don't read tables correctly since the formatting confuses them. Since the chessboard is created from a table format, I thought that may be an issue here. I also am not sure how a screen reader would approach the unicode chess pieces that are added to the board. To address the trouble the screenreader may have with the chessboard, I wondered if there was a way to add alternative text for it, like you would for an image. If it is not possible to add alternate text to the table, the issue could also be fixed by screenshotting the chessboard from the live server preview, and then uploading it to the site as an image with alternative text.

I also wondered how a form would be read by a screenreader - would the reader prompt the user to answer the form fields by posing the quetions asked? This is another section where alternative text could be helpful. I also know that when you inspect a webpage, you can see the comments left in the HTML code. I wondered if leaving comments/pseudocode could be another way to provide alternative text to someone who has a sight disability.

When I created the Google search page replicate this week, I began considering the accessbility of something like this. I am curious how screen readers approach these results pages, because the real HTML for a google page uses a table to show the sublinks under a result, just like I did in this assignment. I wonder if screenreaders are able to read these tables properly, or if it is confusing for the user. Also, I think there are some accessbility concerns to address regarding the page and type colors. The real dark mode in Chrome is more well balanced and contrasted, but my replicate is definitely a bit difficult to read with the bright blue and purple linked text against the charcoal grey background. When I used the "Inspect" tool on a real google search page, there was a little section called "Accessibility" when hovering over each element on the page. The Accessbility section included information about the name of the element, the level of contrast, it's role on the page (i.e., link), and a yes or no as to if it is keyboard focusable. I looked up the meaning of keyboard focusable, and it means that for a page to be fully accessible, it must be able to be completley controlled and engaged with using only a keyboard. Keyboard focusable elements can be interacted with or opened only using a keyboard. This concept is really interesting to me and I hope to learn more about how to design fully accessible webpages thorughout this class.
